# Codebase Deep Dive（代码库深度分析）

## 目标

对用户给定的开源项目或项目内某个文件夹进行“从 0 到可行动”的深入分析，最终输出：
- **项目地图**：目录结构、关键模块、入口点、数据流/控制流线索
- **架构判断**：分层/边界、主要模式与反模式、关键依赖关系
- **依赖与风险审计**：第三方依赖、潜在安全/供应链风险、配置与密钥风险
- **质量与可测试性**：可维护性热点、复杂度/重复、测试现状与缺口
- **行动清单**：按优先级（高/中/低）给出可执行的改造与验证步骤
- **图解表达**：用流程图和架构图可视化核心事件处理逻辑与模块关系

## 何时使用

- 你需要快速接手陌生仓库，搞清“它怎么跑起来、核心在哪、风险在哪”
- 你要对某个子模块做技术尽调或重构评估
- 你希望产出一份可落地的重构/治理计划，而不是泛泛而谈

## 使用方式（强烈建议）

在对话里：
- 指定分析范围：用 `@` 选中目标目录（例如 `@task/` 或整个仓库根目录）
- 说明你的目的：例如“我要加功能/我要重构/我要安全审计/我要做性能优化”

示例：
- `/codebase-deep-dive @task/ 我想把这部分做成可复用模板，并补齐测试`
- `/codebase-deep-dive @. 评估整体架构与风险，给出治理路线图`

## 约束与原则

- **先地图、再结论**：任何架构判断必须引用代码证据（文件路径/符号/片段）
- **少打扰**：默认不改代码；只有用户明确要求时才提出修改建议或 PR 计划
- **可执行**：所有建议都要配“怎么验证”的步骤或命令（按项目栈自适配）
- **不臆测**：遇到不确定信息，用“需要确认/可用证据推断”区分

## 工作流（你必须严格按顺序执行）

### 0) 收集上下文（输入缺失时也要继续）

1. 确认分析范围：用户给的 `@目录` 或默认仓库根目录
2. 识别语言/构建系统：
   - 优先查看：`package.json` / `pyproject.toml` / `requirements.txt` / `pom.xml` / `build.gradle` / `go.mod` / `Cargo.toml` / `.sln` 等
3. 识别运行入口与脚本：
   - 例如：npm scripts、Makefile、Dockerfile、CI 配置

### 1) 建立“项目地图”（必须输出）

输出：
- **Top-level 结构概览**：一级目录含义（用项目证据支撑）
- **关键入口点**：
  - CLI/服务入口、主函数、路由/控制器注册点、任务调度入口等（按栈而定）
- **关键域模型/核心流程**：
  - 以“调用链/数据流”的方式描述 1-3 条最重要的路径

方法建议（按需选择，不要全部一股脑跑）：
- 用快速搜索定位：入口文件、路由注册、依赖注入容器、事件总线、任务 runner
- 只读展开关键文件：读到足以建立链路即可，避免把上下文塞满

### 2) 架构与边界分析（必须输出）

输出：
- **架构风格**：分层/DDD/模块化/插件化/monorepo 等
- **边界与耦合**：
  - 哪些模块高度耦合（以 import/require/引用图为证据）
  - “应该是边界但不清晰”的地方（提出可行的边界切分策略）
- **反模式与技术债热点**（至少 5 条，带证据与影响）

### 3) 依赖与风险审计（必须输出）

输出：
- **依赖概览**：核心依赖与用途（挑 Top 10）
- **风险点**：
  - 明文密钥/Token、危险配置、过宽权限、潜在注入/XSS/路径遍历等（按项目类型）
  - 供应链风险线索（例如 lockfile 缺失、未固定版本、脚本下载执行等）
- **建议的审计命令**（按栈自适配）：
  - Node：`npm audit` / `pnpm audit` / `npx depcheck` 等
  - Python：`pip-audit` / `safety`（如果可用）
  - Java：`mvn -DskipTests=false test` / 依赖树等

注意：如果仓库不是标准应用栈（比如算法题仓库），就把这一节降级为“仓库卫生/许可证/潜在泄露风险”。

### 4) 质量与可测试性评估（必须输出）

输出：
- **可维护性热点**：
  - 超大文件/超长函数/深层嵌套/重复代码（给出定位策略与证据）
- **测试现状**：
  - 是否存在测试框架、覆盖范围大概在哪、最缺的测试类型是什么
- **建议的最小测试策略**：
  - 以“保护核心路径”为目标，给 3-5 条最小可落地测试建议

### 5) 行动清单与路线图（必须输出）

输出必须包含：
- **优先级清单**（高/中/低），每条包含：
  - 问题/机会
  - 影响范围
  - 建议方案
  - 验证方式（测试/命令/检查点）
- **两阶段路线图**：
  - \(0-2\) 天：快速止血/建立护栏（lint、格式化、基本测试、CI）
  - \(1-2\) 周：结构性改造（模块边界、依赖治理、性能/安全专项）

### 6) 图解（必须输出）

输出必须包含两个 Mermaid 图，且每个节点都要尽量映射真实模块/函数/文件：

- **流程图（事件处理逻辑）**：
  - 使用 `flowchart TD` 或 `sequenceDiagram`
  - 至少覆盖：输入事件 -> 核心处理 -> 状态变化/持久化 -> 输出/副作用
  - 至少标注 1-2 个异常分支（如校验失败、重试、回滚）
- **架构图（模块架构逻辑）**：
  - 使用 `graph LR`
  - 至少覆盖：入口层、业务层、数据层、外部依赖（可按项目实际调整）
  - 必须标明主要依赖方向，避免只给目录树

当用户明确要求“画图工具产物”（如 Figma/FigJam）时：
- 先输出 Mermaid 文本图（保证可复制与可审阅）
- 再补充说明如何将图迁移到 Figma/FigJam（如需）

## 最终输出格式（必须严格遵守）

请按以下标题输出（不要改标题顺序）：

1. **Executive Summary（要点）**
2. **Project Map（项目地图）**
3. **Architecture & Boundaries（架构与边界）**
4. **Dependencies & Risks（依赖与风险）**
5. **Quality & Testing（质量与测试）**
6. **Action Plan（行动清单与路线图）**
7. **Diagrams（图解）**
